叫我 ivy。

你在这个项目中的长期工作规则如下：

一、默认协作方式
1. 默认直接执行，减少过程性叙述，不输出长篇规划。
2. 除非存在高风险、关键分叉、信息缺失或可能造成不可逆后果，否则不要频繁向我确认。
3. 优先给结果，其次再用简短几句话说明你做了什么。
4. 不要复述我的需求，不要重复历史上下文，不要输出冗长总结。
5. 每一轮只聚焦当前目标，不主动扩展无关优化。
6. 输出默认简洁，除非我明确要求详细解释、方案对比、教学式展开或完整分析。

二、节省额度与减少卡顿
1. 任务较小时直接执行，不先写大段计划。
2. 中等任务先用一句话说明计划，然后直接执行。
3. 较大任务先给精简步骤清单，等我确认后分阶段执行。
4. 能并行处理的子任务就并行处理，但最终只汇总关键结论，不展示冗长中间过程。
5. 不重复读取无关文件，不通读整个项目，优先读取与当前任务最相关的文件。
6. 如果对话已经很长，优先基于项目记忆文件重新建立上下文，而不是依赖整段聊天历史。
7. 如果历史上下文已经影响效率，主动建议切换到新对话，并先更新项目交接文档。
8. 避免输出过长的分析、过多的备选方案、重复性的解释和无必要的代码摘录。
9. 除非我明确要求，否则不要展示大段命令输出、完整日志或大块文件内容。
10. 保持回答短而有用，优先保留结论、关键修改、风险、下一步。

三、项目长期记忆规则
1. 不把长期记忆依赖在聊天记录里，而是落到项目文件里。
2. 该项目固定维护以下记忆文件：
   AGENTS.md：协作规则
   HANDOFF.md：项目交接状态
   DECISIONS.md：关键决策与约束
3. 新对话开始时，优先读取 AGENTS.md、HANDOFF.md、DECISIONS.md，再继续执行。
4. 不要假设不同对话自动共享完整记忆；应通过这些项目文件实现共享状态。
5. 保持这些文件简洁，只保留当前有效信息，不写流水账。

四、何时建议切换到新对话
当出现以下任一情况时，主动建议我切换到新对话：
1. 当前阶段已经完成，准备进入新阶段。
2. 任务方向明显切换，例如从开发切到调试、从实现切到部署、从编码切到重构。
3. 对话明显变长，回答开始变慢，或开始频繁重复历史内容。
4. 当前信息已经充分落盘到代码和记忆文件中，不再需要依赖旧聊天历史。

五、工作区整洁规则
1. 保持代码库整洁，不创建无用临时文件，不保留死代码，不新增无必要目录。
2. 临时产物仅在确有必要时创建，并尽量集中、易清理。
3. 修改完成后，优先留下清晰、可继续协作的状态，而不是留下大量临时痕迹。

六、当我说“生成项目交接文档”时，你必须执行固定流程
当我输入“生成项目交接文档”时，不要写泛泛总结，必须执行以下步骤：

步骤 1：先扫描当前有效状态
只基于当前项目中真实有效的信息生成交接内容，包括：
- 当前已存在的代码与文件
- 本轮已经完成的工作
- 当前未完成但已明确的任务
- 关键决策、约束、风险、依赖
- 下一步最适合立即执行的动作

步骤 2：更新或生成以下文件
- HANDOFF.md
- DECISIONS.md
如果没有新决策，可保留 DECISIONS.md 原有内容并只做必要补充。

步骤 3：HANDOFF.md 必须包含以下固定结构
# Current Goal
当前总目标，1 到 3 句。

# Current Stage
当前处于哪个阶段，例如需求、实现、调试、重构、测试、部署。

# Done
本阶段已完成事项，只写有效结果，不写过程流水账。

# In Progress
当前正在进行但未完成的事项。

# Blockers
阻塞点、风险点、待确认事项；如果没有，明确写 None。

# Key Files
当前最关键的文件路径，并说明每个文件的作用。

# How To Continue
新对话接手时应该先看什么、做什么。

# Next Action
下一位 Codex 应立即执行的第一步，要求具体、可操作、可验证。

# Run / Verify
如有需要，写出最关键的运行、测试、验证命令。

# Notes
只保留对下一步真正有帮助的补充说明，避免重复。

步骤 4：DECISIONS.md 必须记录
- 已确认的技术方案
- 明确不采用的方案
- 关键约束
- 已确定的命名、目录、接口、风格约定
- 后续对话不应反复推翻的决定

步骤 5：生成交接文档后，你的回复必须做到
1. 先告诉我交接文件已更新。
2. 用不超过 5 句话概括项目当前状态。
3. 给出一条建议：现在继续留在本对话，还是切到新对话更合适。

七、新对话接手规则
当我在新对话中说明“继续这个项目”时，你必须：
1. 先读取 AGENTS.md、HANDOFF.md、DECISIONS.md。

3. 用不超过 5 句话总结当前状态。

5. 如果发现交接文档与代码现状不一致，先指出差异，再按代码真实状态继续。

5. 如果我只发一句短指令，优先理解为要你直接开工，而不是先讨论。

目标是以最少的上下文消耗，维持稳定、连续、高效的项目推进。长期状态保存在项目文件中，而不是寄希望于单个对话永久记住全部历史。
